# 438C 文档生成与知识图谱系统 - 全站部署验收清单

## 1. 适用范围

本清单用于以下部署方式的上线前验收：

- `docker compose` 一键部署
- Linux 离线部署
- Windows 离线部署

当前项目的核心服务包括：

- `frontend`：前端 Nginx 静态站点
- `backend`：Flask API 服务
- `postgres`：关系型数据库
- `redis`：缓存与 Celery Broker
- `chromadb`：向量数据库
- `neo4j`：图数据库
- `minio`：对象存储
- `celery-worker`：异步任务 Worker

## 2. 验收目标

验收通过标准：

- 所有核心容器或服务可启动且健康
- 前端页面可访问，主要按钮、菜单、路由显示正常
- 后端核心 API 可返回成功结果
- 知识库导入、知识图谱、问答、文档生成与下载可完成闭环
- Neo4j、MinIO、ChromaDB、PostgreSQL、Redis 连通正常
- 关键日志中无阻塞性异常

## 3. 验收前准备

### 3.1 环境确认

- 操作系统满足部署文档要求
- Docker / Docker Compose 可正常使用
- 端口未被占用：`80`、`5000`、`5432`、`6379`、`7474`、`7687`、`8000`、`9000`、`9001`
- 已准备 `.env` 或等效环境变量
- 已准备至少 1 份测试模板文件，建议使用 `438c` 目录中的样例文档

### 3.2 必填配置

重点检查以下配置项：

- `POSTGRES_DB`
- `POSTGRES_USER`
- `POSTGRES_PASSWORD`
- `REDIS_PASSWORD`
- `MINIO_ACCESS_KEY`
- `MINIO_SECRET_KEY`
- `MINIO_BUCKET`
- `NEO4J_URI`
- `NEO4J_USER`
- `NEO4J_PASSWORD`
- `LLM_PROVIDER`
- `LLM_API_BASE`
- `LLM_API_KEY`
- `LLM_MODEL`

### 3.3 启动命令

```bash
docker compose up -d --build
```

查看状态：

```bash
docker compose ps
```

查看日志：

```bash
docker compose logs -f backend
docker compose logs -f frontend
docker compose logs -f celery-worker
```

## 4. 基础设施验收

### 4.1 容器状态

检查项：

- `postgres` 为 `healthy`
- `redis` 为 `healthy`
- `chromadb` 为 `healthy`
- `neo4j` 为 `healthy`
- `minio` 为 `healthy`
- `backend` 为 `Up`
- `frontend` 为 `Up`
- `celery-worker` 为 `Up`

通过标准：

- 所有服务启动成功，无反复重启

失败排查：

- `docker compose logs --tail=200 <service>`

### 4.2 PostgreSQL

检查命令：

```bash
docker exec -it doc-gen-postgres psql -U docgen -d doc_gen -c "\dt"
```

通过标准：

- 能连接数据库
- 能看到系统表或业务表

### 4.3 Redis

检查命令：

```bash
docker exec -it doc-gen-redis redis-cli -a redis123456 ping
```

通过标准：

- 返回 `PONG`

### 4.4 ChromaDB

检查命令：

```bash
curl http://localhost:8000/api/v1/heartbeat
```

通过标准：

- 返回心跳响应

### 4.5 Neo4j

检查命令：

```bash
curl http://localhost:7474
```

通过标准：

- 浏览器可访问 `http://localhost:7474`
- Bolt 端口 `7687` 可连通

建议人工检查：

- 使用 `neo4j / 配置密码` 登录 Browser
- 执行 `MATCH (n) RETURN count(n);`

### 4.6 MinIO

检查命令：

```bash
curl http://localhost:9000/minio/health/live
curl http://localhost:9001
```

通过标准：

- `9000` 健康检查可返回成功
- `9001` 控制台可访问

建议人工检查：

- 登录 MinIO 控制台
- 检查默认 bucket 是否存在或可自动创建

## 5. 后端 API 验收

### 5.1 基础接口

检查命令：

```bash
curl http://localhost:5000/api/project/projects
curl http://localhost:5000/api/438c/document-types
curl http://localhost:5000/api/knowledge/graph
```

通过标准：

- 三个接口均返回 `200`
- 返回体为合法 JSON

### 5.2 项目管理

创建项目：

```bash
curl -X POST http://localhost:5000/api/project/projects \
  -H "Content-Type: application/json" \
  -d "{\"name\":\"部署验收项目\",\"system_name\":\"438C系统\",\"organization\":\"测试单位\"}"
```

通过标准：

- 返回新建项目 ID
- 项目列表中可查到该项目

### 5.3 知识库创建

创建知识库：

```bash
curl -X POST http://localhost:5000/api/knowledge/bases \
  -H "Content-Type: application/json" \
  -d "{\"name\":\"验收知识库\",\"description\":\"部署验收使用\"}"
```

通过标准：

- 返回知识库 ID
- `GET /api/knowledge/bases` 中可查询到

## 6. 核心业务链路验收

### 6.1 模板导入与解析

检查项：

- 导入 `438c` 目录中的 `doc` / `docx` 模板
- 文件列表可见
- 文件预览接口能返回正文

推荐路径：

1. 前端进入“知识库管理”
2. 创建知识库
3. 上传 `438c` 目录中的样例文件
4. 打开文件预览

通过标准：

- 上传成功
- 文件状态正常
- 预览中有可读正文

### 6.2 语义检索

检查命令：

```bash
curl -X POST http://localhost:5000/api/knowledge/bases/1/search \
  -H "Content-Type: application/json" \
  -d "{\"query\":\"软件需求规格说明需要哪些章节\",\"top_k\":5}"
```

通过标准：

- 返回检索结果数组
- 结果中包含 `score`、`content` 等关键字段

### 6.3 知识图谱

接口检查：

```bash
curl http://localhost:5000/api/knowledge/graph
```

通过标准：

- 返回 `entities`、`relations`、`stats`
- `stats.storage.neo4j.enabled=true`
- `stats.storage.neo4j.connected=true` 或至少本地 JSON 可回退

前端检查：

1. 打开 `http://localhost/#/knowledge-graph`
2. 检查页面是否正常渲染
3. 检查是否能看到节点、连线、图例、统计卡片

通过标准：

- 页面可打开
- 可拖拽节点
- 可缩放
- 节点文字可见

### 6.4 智能问答

检查命令：

```bash
curl -X POST http://localhost:5000/api/question/ask \
  -H "Content-Type: application/json" \
  -d "{\"question\":\"438C规定了哪些文档类型？\",\"top_k\":5}"
```

通过标准：

- 返回回答文本
- 无 `500` 错误

前端检查：

- 问答页可正常输入问题
- 语音输入按钮可见
- 浏览器授权麦克风后可以把语音转成文本

备注：

- 语音输入依赖浏览器 Web Speech API，建议使用最新版 Chrome / Edge

### 6.5 文档生成

检查路径：

1. 前端创建生成文档
2. 执行整篇生成或章节生成
3. 查看章节内容是否写入

接口示例：

```bash
curl -X POST http://localhost:5000/api/generation/documents \
  -H "Content-Type: application/json" \
  -d "{\"project_id\":1,\"document_type\":\"SRS\",\"title\":\"部署验收SRS\"}"
```

通过标准：

- 文档创建成功
- 生成任务执行成功
- 章节内容可保存与查看

### 6.6 文档导出与下载

检查项：

- 点击前端导出按钮
- 或调用导出接口
- 下载 `docx` 成品

通过标准：

- 导出成功
- 浏览器能下载文件
- 文件可用 Word 正常打开

## 12. 推荐验收顺序

建议按以下顺序执行，效率最高：

1. `docker compose up -d --build`
2. 检查所有服务健康状态
3. 校验 `backend` 三个基础接口
4. 打开前端首页和知识图谱页
5. 创建项目和知识库
6. 上传样例模板并预览
7. 执行一次语义搜索
8. 执行一次问答
9. 执行一次文档生成
10. 执行一次导出下载
11. 检查 Celery、Neo4j、MinIO、Backend 日志
12. 填写最终验收结论表
